# 设计：CodexPlusPlus 桌面端模型后缀提示

> 日期：2026-06-24  
> 状态：已批准，待实现  
> 依赖：阶段一 model catalog 原型后端已完成（`model_suffix` 模块 + `apply_model_catalog_to_config`）

---

## 1. 背景

阶段一已在后端实现 `model_list` 后缀语法（如 `deepseek-v4-pro[1M]`），应用 relay profile 时会自动生成 `model_catalog_json` 并写入 config.toml 指针，使 codex 客户端按模型识别真实上下文窗口。

但前端 `codex-plus-manager` 的“配置模型”和“模型列表”输入框目前没有任何提示说明支持后缀，用户不知道如何填写。本设计目标是以**最小改动**在前端暴露这一能力。

---

## 2. 目标

- 让用户知道可以在“配置模型”和“模型列表”中给模型名加后缀来指定上下文窗口。
- 保持现有输入框组件和交互不变。
- 前端不解析、不校验、不显示解析后的窗口值（决策权完全交给用户）。

---

## 3. 范围

**在范围内：**
- 更新 `apps/codex-plus-manager/src/App.tsx` 中两个模型相关输入框的 placeholder。
- 在两个输入框下方各新增一行 hint 文案。

**不在范围内：**
- 结构化 model_list 编辑器（阶段三）。
- 后缀实时高亮、校验、窗口可视化。
- 后端逻辑改动（已完成后端无需再改）。
- 新增依赖或组件。

---

## 4. 设计

### 4.1 修改位置

文件：`apps/codex-plus-manager/src/App.tsx`

#### 4.1.1 “配置模型”字段

当前代码位置约在 3788 行附近：

```tsx
<Field className="relay-field-config-model" label="配置模型">
  <Input
    value={profile.model}
    onChange={(event) => updateDraft({ model: event.currentTarget.value })}
    placeholder="留空使用默认值"
  />
</Field>
```

改为：

```tsx
<Field className="relay-field-config-model" label="配置模型">
  <Input
    value={profile.model}
    onChange={(event) => updateDraft({ model: event.currentTarget.value })}
    placeholder="例如 deepseek-v4-pro 或 deepseek-v4-pro[1M]"
  />
  <p className="field-hint">
    支持在模型名后加 <code>[1M]</code>、<code>[200K]</code> 或 <code>[1000000]</code> 指定上下文窗口；不写则使用 Codex 默认长度。
  </p>
</Field>
```

#### 4.1.2 “模型列表”字段

当前代码位置约在 3898 行附近：

```tsx
<Field className="relay-field-model-list" label="模型列表">
  <div className="relay-model-list-tools">
    <Textarea
      value={profile.modelList}
      onChange={(event) => updateDraft({ modelList: event.currentTarget.value })}
      placeholder="每行一个模型，例如 qwen3-coder"
    />
    {/* ... */}
  </div>
</Field>
```

改为：

```tsx
<Field className="relay-field-model-list" label="模型列表">
  <div className="relay-model-list-tools">
    <Textarea
      value={profile.modelList}
      onChange={(event) => updateDraft({ modelList: event.currentTarget.value })}
      placeholder="每行一个模型，例如 qwen3-coder 或 deepseek-v4-pro[1M]"
    />
    {/* ... */}
  </div>
  <p className="field-hint">
    每行一个模型，支持在模型名后加 <code>[1M]</code>、<code>[200K]</code> 或 <code>[1000000]</code>；不写后缀的模型使用 Codex 默认长度。
  </p>
</Field>
```

### 4.2 样式

使用项目中已有的 `.field-hint` 类（定义在 `apps/codex-plus-manager/src/styles.css:2100`），无需新增 CSS。

```css
.field-hint {
  margin: -4px 0 12px;
  color: hsl(var(--muted-foreground));
  font-size: 12px;
}
```

两个 hint 段落均使用 `<p className="field-hint">...</p>`。

### 4.3 文案约定

- 后缀示例统一使用 `[1M]`、`[200K]`、`[1000000]`。
- 明确说明“不写则使用 Codex 默认长度”，避免用户以为必须写后缀。

---

## 5. 数据流

前端仅做文案提示，不增加新的状态或 API 调用。

```
用户输入带后缀的模型名
        ↓
保存/应用 profile（现有流程）
        ↓
后端 parse_model_suffix / collect_catalog_entries / build_model_catalog_json
        ↓
generate catalog + 写 config.toml 指针（现有流程）
```

## 6. 测试计划

### 6.1 手工测试

1. 启动 `codex-plus-manager`（`npm run dev` 或已构建的 app）。
2. 打开一个 Relay Profile 编辑页。
3. 在“配置模型”输入 `deepseek-v4-pro[1M]`，保存。
4. 在“模型列表”输入：
   ```
   deepseek-v4-pro[1M]
   claude-sonnet-4[200K]
   qwen3-coder
   ```
   保存并应用 profile。
5. 检查 `~/.codex/config.toml` 包含 `model_catalog_json = "model-catalogs/<profile-id>.json"`。
6. 检查 `~/.codex/model-catalogs/<profile-id>.json` 中：
   - `deepseek-v4-pro` 的 `context_window` 为 `1000000`
   - `claude-sonnet-4` 的 `context_window` 为 `200000`
   - `qwen3-coder` 的 `context_window` 为 Codex 默认值或顶层 fallback
7. 运行 `codex debug models` 验证窗口正确。

## 8. 风险与回退

| 风险 | 应对 |
|------|------|
| hint 文案在不同主题下对比度不足 | 使用项目现有 muted foreground 颜色变量 |
| placeholder 过长在某些窗口宽度下显示不全 | placeholder 保持简洁，完整说明放 hint |
| 用户误以为前端会校验后缀 | hint 文案不承诺校验，保持“支持写”而非“必须写对” |
